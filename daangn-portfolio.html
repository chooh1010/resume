PoppingOps — 운영 자동화 플랫폼 관점의 모니터링/알림 설계

개인 프로젝트 (2026.04) · Bash · Railway · Docker · GitHub Actions · GitHub

1. 로컬 PC 의존 감시를 운영 자동화 가능한 상시 모니터링 구조로 전환

Problem
  • t2.micro 환경에서 Prometheus/Grafana를 상시 운영하기 어려워 로컬 PC 기반 모니터링에 의존
  • PC가 꺼지면 모니터링과 알림이 함께 중단돼 운영 자동화가 불가능한 구조
Analyze
  • 핵심은 "어디서나 돌아가는 가벼운 감시 경로"와 "사람이 읽을 수 있는 보고 경로"를 분리하는 것이라고 판단
  • 정기 수집, 상태 판단, 알림, 요약 보고를 하나의 경로로 묶으면 비용과 실패 범위가 커짐
Action
Railway Container └─ health-check.sh ├─ 10분마다 SSH로 actuator / node-exporter / mysqld-exporter 수집 ├─ snapshot 저장 ├─ counter delta로 응답시간 / 에러율 / 트래픽 계산 └─ WARN / CRITICAL / 복구 알림 전송
  • Railway 컨테이너가 10분마다 메트릭을 수집하고 snapshot 간 delta로 rate 지표를 계산
  • 수동 확인 없이 최대 10분 이내 이상 징후를 인지할 수 있는 운영 구조로 전환
Result
모니터링 의존성로컬 PC→Railway 상시 실행
장애 인지수동 확인→최대 10분 이내 자동 알림

2. 정기 감지, 권장 조치, 보고 경로를 분리해 운영 비용과 실패 범위를 축소

Analyze
검토 방식검토 결과판단
정기 감지까지 LLM 사용 정상 상태에서도 반복 비용이 발생하고, LLM 장애가 감지 경로 장애로 이어짐 기각
모든 권장 조치에 LLM 사용 반복적인 같은 알림에도 같은 조치를 계속 생성해 비용과 지연이 누적 기각
규칙 기반 감지 +
runbook 우선 +
필요 시만 LLM		 정기 메트릭 수집과 임계값 비교는 bash가 처리하고, LLM은 사용자 질문 / 요약 보고 / 예외 fallback에만 사용 채택
Action
Detection Layer └─ bash 임계값 비교로 상태 전이 판단 Recommendation Layer ├─ JSON runbook 우선 적용 └─ 미매칭 시에만 LLM fallback Analysis Layer ├─ 사용자 질문 응답 ├─ Daily Summary / Full Report └─ 운영자용 해석 메시지
  • 정기 감지와 기본 장애 알림은 bash가 처리하고, 권장 조치는 JSON runbook을 우선 적용
  • LLM은 사용자 질문, Daily Summary / Full Report, runbook 미매칭 alert fallback처럼 해석이 필요한 경우에만 사용
  • 운영 자동화 경로와 해석 경로를 분리해 비용과 실패 범위를 축소
Result
정상 상태 토큰 비용반복 소모 가능→0원
권장 조치 생성매번 LLM 호출→runbook 우선, 미매칭만 fallback

3. 모니터링 자체 실패를 별도 추적해 감지 불능 상태를 정상으로 오인하지 않도록 설계

Problem
  • SSH 실패, exporter 응답 누락, stale snapshot이 발생해도 이를 별도로 드러내지 않으면 "정상"으로 오인할 수 있음
Action
health-check.sh / heartbeat-context.sh ├─ SSH 연결 확인 → 실패 시 연결 실패 카운트 증가 ├─ 메트릭 수집/검증 → 파싱 실패 또는 필수 값 누락 시 resource_parse 카운트 증가 └─ snapshot 저장 → 저장 실패 시 snapshot_write 카운트 증가 [유형별 2회 연속 실패] → "모니터링 장애 알림" snapshot freshness 판정 ├─ <20분: OK ├─ 20~30분: WARN └─ >30분: CRITICAL 필수 메트릭/핵심 값 검증 ├─ app/node/mysql metrics 누락 ├─ 핵심 값 비어 있음/0 이하 └─ 위 경우 모두 resource_parse로 처리하고 snapshot 갱신 중단
  • 운영 대상 서버 장애와 운영 파이프라인 장애를 분리해 알림
  • stale snapshot과 불완전한 수집 결과가 정상처럼 읽히지 않도록 방어
