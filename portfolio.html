1. 댓글 트리 조회 병목 해소 — Cache-Aside + 이벤트 기반 무효화

1. 시각화
Client → [Spring Boot] ↓ cache.get(postId) └ HIT → 캐시 데이터 반환 + 좋아요/개인 반응 별도 합성 ┌ MISS → 재귀 CTE 쿼리 1회 실행 → 캐시 저장 → 반환 [댓글 작성/삭제] ↓ DB 커밋 완료 ↓ @TransactionalEventListener(AFTER_COMMIT) ↓ cache.evict(postId)
2. 문제
  • 게시글 상세 조회 시 댓글 첫 페이지가 함께 내려가는 구조. 댓글은 대댓글을 포함한 트리라 재귀 CTE 쿼리가 사용됨
  • 동일한 CTE가 매 조회마다 반복 실행되어 병목 발생 — JMeter 100 VUser, 10분간 6,642회 실행 확인
3. 해결과정 — 왜 Cache-Aside를 선택했는가?
검토 방식검토 결과판단
Write-Through 트리 구조 특성상 댓글 작성 시 캐시 갱신 비용이 크고, 캐시 로직이 DB보다 복잡해짐 기각
부분 무효화 변경 댓글만 교체하려면 트리 재조립 로직이 캐시 레이어에 침투, 복잡도 높음 기각
Cache-Aside 안정적인 공통 데이터(트리 구조)만 캐시, 좋아요 수·개인 반응은 조회 시점에 별도 합성 채택
  • 댓글 작성/삭제 시 @TransactionalEventListener(AFTER_COMMIT)으로 커밋 이후에만 evict → stale 데이터 캐싱 가능성 제거
  • Caffeine maximumSize=500, expireAfterWrite=10m, recordStats로 히트율 모니터링
4. 결과/성과
인기 게시글 댓글 조회152 ms→43 ms
CTE 실행 횟수 (10분)6,642건→458건
캐시 히트율—→평균 ~80%
JMeter 100 VUser, 10분, 게시글 100만 건·댓글 1,000만 건, EC2 실서버 기준

2. 캐시 스탬피드 방지 — Caffeine 원자적 로딩

1. 시각화
// Before: get과 put이 분리 → 캐시 스탬피드 발생 CommentPage page = cache.getIfPresent(postId); if (page == null) { page = buildCommentPage(postId); // 여러 스레드가 동시 진입 cache.put(postId, page); } // After: 원자적 로딩 → 동일 postId에 대해 1회만 실행 CommentPage page = cache.get(postId, key -> buildCommentPage(key));
2. 문제
  • 캐시가 evict된 직후 동일 postId에 동시 요청이 몰리면 재귀 CTE가 10~17건 중복 실행
  • 기존 구조는 cache.get → miss → build → cache.put으로, get과 put이 분리되어 여러 스레드가 동시에 DB 재생성 경로로 진입 (캐시 스탬피드)
  • JMeter(HTTP 환경 재현) + CountDownLatch 기반 단위 테스트(원자성 검증)로 재현
3. 해결과정 — 왜 Caffeine cache.get(key, Callable)을 선택했는가?
검토 방식검토 결과판단
전역 락 (synchronized) 서로 다른 postId 요청까지 함께 대기시켜 처리량 저하가 큼 기각
ConcurrentHashMap
per-key 락		 락 객체 생성·해제 관리 복잡도와 메모리 부담 기각
Caffeine
cache.get(key, Callable)		 동일 key에 대한 재생성을 1회로 제한, 다른 key는 병렬 처리 유지 채택
4. 결과/성과
evict 직후 50건 동시 요청CTE 10~17건 중복→CTE 1건만 실행
  • 나머지 49건은 캐시 히트로 처리, key 단위 동작이므로 다른 게시글은 병렬 처리 유지
  • 한계: 단일 JVM 환경에서만 유효 — 다중 인스턴스 환경에서는 분산 락(Redis 등) 기반 확장 필요
  • 트레이드오프: 캐시 미스 시 첫 요청 완료까지 후속 요청 대기 (테스트 기준 최대 315ms)

3. 좋아요 동시 요청 멱등 처리 — ON DUPLICATE KEY UPDATE

1. 시각화
// INSERT 멱등 처리 (affected rows = 1이면 신규 삽입) @Query(value = "INSERT INTO likes (user_id, target_type, target_id) " + "VALUES (:userId, :targetType, :targetId) " + "ON DUPLICATE KEY UPDATE id = id", nativeQuery = true) int insertIgnore(Long userId, String targetType, Long targetId); // 실제 insert된 경우에만 count 증가 int affected = likeRepository.insertIgnore(userId, targetType, targetId); if (affected > 0) { likeRepository.incrementCount(targetType, targetId, +1); }
2. 문제
  • 토글 기반 좋아요 처리에서 check-then-act 경쟁 상태 발생
  • 여러 스레드가 동시에 "좋아요가 없다"고 판단 → insert 경쟁 → 한 요청만 성공, 나머지는 유니크 제약 예외
  • JMeter(HTTP 환경 재현) + CountDownLatch 기반 단위 테스트(원자성 검증)로 재현
3. 해결과정 — 왜 ON DUPLICATE KEY UPDATE를 선택했는가?
검토 방식검토 결과판단
비관적 락 작업 전체를 직렬화해 처리량 급감, 데드락 위험 기각
saveAndFlush +
예외 처리		 트랜잭션이 rollback-only 상태 → UnexpectedRollbackException 발생 기각
ON DUPLICATE KEY
UPDATE id = id +
affected rows		 실제 insert 성공 시에만 count 반영, 예외 없이 멱등 처리 채택
4. 결과/성과
동일 사용자 동시 20 VUser유니크 제약 예외 다수→예외 0건, 멱등 처리
  • 중복 요청에도 row 생성과 count 증가가 각각 1회만 반영
  • 삭제도 DELETE 영향 row 수가 1인 경우에만 count 감소 — 동일한 멱등 구조 적용

4. 대용량 집계 병목 제거 — 쿼리 분리 + 비정규화

1. 시각화
Before (단일 쿼리) SELECT COUNT(*), EXISTS(...) ← likes 450만 건 풀스캔 FROM likes WHERE target_type = ? AND target_id = ? ↑ UK 선두 컬럼이 user_id라 인덱스 레인지 스캔 불가 After (분리) [1] 집계값 ← comment.like_count (비정규화 칼럼, 인덱스 불필요) [2] 개인 반응 ← SELECT ... FROM likes WHERE user_id = ? AND target_id = ? ↑ UK 선두 칼럼 user_id 활용, 인덱스 레인지 스캔
2. 문제
  • likes 450만 건 적재 후 부하 테스트 → 에러율 15.53%, 평균 응답시간 10,460ms까지 악화
  • Slow Query Log 확인 결과, 댓글 반응 집계 쿼리가 likes 테이블을 전체 스캔
  • 원인: likeCount(집계)와 likedByMe(개인화)를 한 쿼리에서 처리 → UK 선두 칼럼이 user_id라 Leftmost Prefix Rule에 의해 인덱스 레인지 스캔 불가
3. 해결과정 — 왜 쿼리 분리 + 비정규화를 선택했는가?
검토 방식검토 결과판단
집계 전용
인덱스 추가		 에러 해소, 응답시간 정상화 — 그러나 별도 인덱스 유지 비용, 데이터 커질수록 부담 한계
쿼리 구조 분리 +
비정규화 칼럼		 집계는 comment 테이블의 비정규화 칼럼, 개인 반응만 likes에서 조회 — 별도 인덱스 불필요 채택
  • 비정규화로 인한 count 불일치 위험은 재집계 배치로 보완
  • 데이터 규모 확장 시 증분 집계나 CDC 기반 구조로 확장 가능하도록 고려

팀 프로젝트 (4명) · 2024.01 ~ 2024.04 · EC2 실 배포 · 본인 담당: 백엔드 전체

1. Spring REST Docs 기반 API 문서화 체계 구축

1. 시각화
테스트 코드 작성 ↓ ./gradlew test ↓ asciidoctor (snippets → HTML 문서) ↓ copyDocument (static/docs에 배치) ↓ 테스트 통과 시에만 문서 갱신 → 문서 = 항상 최신 상태
2. 문제
  • Swagger는 문서용 어노테이션이 프로덕션 코드에 섞이고, 코드 변경 시 문서와 실제 동작이 어긋날 수 있음
3. 해결과정
  • Spring REST Docs 도입, test → asciidoctor → copyDocument 흐름의 문서 생성 파이프라인 구성
  • 테스트 통과 시에만 문서가 생성되도록 만들어 문서 = 항상 최신 상태 보장
  • RestDocsTest, RestDocsConfig, ApiDocumentUtils를 분리해 공통 문서 설정 재사용
  • @WithAuthUser + Mock SecurityContext를 적용해 Firebase 인증 환경에서도 테스트와 문서화를 일관되게 수행
4. 결과/성과
  • 테스트와 문서가 강하게 연결된 문서화 체계 구축
  • 프로덕션 코드 오염 없이 API 문서 품질 유지

2. GitHub Actions + Jib 기반 CI/CD 자동화

1. 시각화
main push ↓ GitHub Actions └ REST Docs 빌드 └ Jib 이미지 생성 → Docker Hub push ┌ SSH → EC2 docker-compose up -d
2. 문제
  • 로컬 빌드, Docker 이미지 생성, Docker Hub push, EC2 재배포까지 수동으로 처리
3. 해결과정
  • Jib를 사용해 Dockerfile 없이 이미지 빌드 — 레이어 캐싱으로 빌드 효율 향상
  • Docker Hub Access Token, SSH 키 등 민감 정보는 GitHub Secrets로 분리 관리
4. 결과/성과
  • 문서 생성, 이미지 빌드, 배포가 하나의 파이프라인으로 연결
  • 수동 배포 절차를 제거해 팀이 기능 개발과 검증에 집중할 수 있도록 지원
