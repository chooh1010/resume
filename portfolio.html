Problem

  • 동일한 CTE가 반복 실행되어 병목 발생 — JMeter 100 VUser, 10분간 6,642회 실행 확인
Analyze
검토 방식검토 결과판단
Write-Through 쓰기 직후 즉시 최신 데이터를 보장할 필요가 없는 도메인. 동시 쓰기 시 느린 재조립이 최신 캐시를 덮어쓰는 정합성 문제도 존재 기각
부분 무효화 CommentResponse가 record(불변)라 노드 하나를 바꾸면 루트까지 경로 전체를 재생성해야 함. 전체 재조립과 비용 차이 없음 기각
Cache-Aside 안정적인 공통 데이터(트리 구조)만 캐시, 좋아요 수·개인 반응은 조회 시점에 별도 합성 채택
Action
Client → [Spring Boot] ↓ cache.get(postId) └ HIT → 캐시 데이터 반환 + 좋아요/개인 반응 별도 합성 ┌ MISS → 재귀 CTE 쿼리 1회 실행 → 캐시 저장 → 반환 [댓글 작성/삭제] ↓ DB 커밋 완료 ↓ @TransactionalEventListener(AFTER_COMMIT) ↓ cache.evict(postId)
  • Cache-Aside 패턴으로 댓글 첫 페이지(page=0)를 postId 키로 캐싱
  • 캐시에는 트리 구조만 저장, likeCount·dislikeCount는 조회 시 Comment 테이블 IN절로 합성 → 좋아요 발생 시 캐시 evict 불필요
  • 같은 게시글이라도 사용자마다 likedByMe/dislikedByMe가 달라 공통 캐시 단일 반환 불가 → likedByMe=false 고정 저장, 조회 시 현재 사용자의 reaction을 IN() 배치 쿼리 1회로 합성
  • 트랜잭션 안에서 evict하면 커밋 전에 다른 스레드가 캐시 미스를 보고 아직 커밋되지 않은 이전 상태를 읽어 캐시에 올릴 수 있음 → @TransactionalEventListener(AFTER_COMMIT)으로 커밋 완료 후 evict
  • Caffeine maximumSize=500, expireAfterWrite=10m, recordStats로 히트율 모니터링

2. 캐시 스탬피드 방지 — Caffeine 원자적 로딩 Blog

Problem
  • 캐시가 evict된 직후 동일 postId에 동시 요청이 몰리면 재귀 CTE가 중복 실행 — JMeter 50건 동시 요청에서 10~17건, CountDownLatch 기반 단위 테스트에서 50건 전부 중복 실행 확인
  • 기존 구조는 cache.get → miss → build → cache.put으로, get과 put이 분리되어 여러 스레드가 동시에 DB 재생성 경로로 진입
// Before: get과 put이 분리 → 캐시 스탬피드 발생 CommentPageResponse cached = cache.get(postId, CommentPageResponse.class); if (cached != null) { return cached; } CommentPageResponse fresh = buildCommentPage(postId, 0); // 여러 스레드가 동시 진입 cache.put(postId, fresh);
Analyze
검토 방식검토 결과판단
전역 락 (synchronized) 서로 다른 postId 요청까지 함께 대기시켜 불필요한 처리량 저하 기각
ConcurrentHashMap
per-key 락 직접 구현		 캐시와 별도로 락 생명주기를 관리하는 복잡성이 추가됨 기각
Caffeine
cache.get(key, Callable)		 내부적으로 per-key 원자적 로딩을 제공. 락 생명주기를 캐시 엔트리에 맞춰 자동 관리 채택
Action
// After: 원자적 로딩 → 동일 postId에 대해 1회만 실행 return cache.get(postId, () -> readOnlyTx.execute(status -> buildCommentPage(postId, 0)));
  • cache.get(key, Callable)로 동일 key에 대한 CTE 실행을 1회로 제한, 다른 key는 병렬 처리 유지
  • Callable 내부에서 TransactionTemplate으로 트랜잭션을 감싸, 향후 refreshAfterWrite 전환 시 Caffeine 백그라운드 스레드(ForkJoinPool)에서 실행되더라도 트랜잭션 컨텍스트 보장
Result
evict 직후 JMeter 50 VUser 동시 요청 (ramp 0s)CTE 10~17건 중복→CTE 1건만 실행
  • 나머지 49건은 캐시 히트로 처리, key 단위 동작이므로 다른 게시글은 병렬 처리 유지
  • CountDownLatch 기반 단위 테스트(50 스레드 동시 실행)로 CTE 실행 1회 고정 검증 통과
  • 트레이드오프: 캐시 미스 시 첫 요청 완료까지 후속 요청 대기 (테스트 기준 최대 315ms)

3. 좋아요 동시 요청 멱등 처리 — ON DUPLICATE KEY UPDATE Blog

Problem
  • 토글 기반 좋아요 처리에서 check-then-act 경쟁 상태 발생
  • 여러 스레드가 동시에 "좋아요가 없다"고 판단 → insert 경쟁 → 한 요청만 성공, 나머지는 유니크 제약 예외 — CountDownLatch 기반 단위 테스트로 동일 사용자 20건 동시 요청 시 17건 예외, JMeter 20 VUser 동시 요청에서도 예외 발생 확인
Analyze
검토 방식검토 결과판단
비관적 락 Like row가 없으면 잠글 대상이 없어 Post에 락을 걸어야 함. 인기 게시글이 hot row가 되어 전체 직렬화로 처리량 급감 기각
saveAndFlush +
예외 처리		 트랜잭션 내 SQL 실패 시 Spring이 rollback-only로 마킹 → catch 후 정상 반환해도 커밋 시점에 UnexpectedRollbackException 발생 기각
ON DUPLICATE KEY
UPDATE id = id +
affected rows		 사전 조회 없이 insert 먼저 시도, 중복 시 no-op. affected rows로 실제 insert 여부를 판단해 count 증감 채택
Action
// INSERT 멱등 처리 (affected rows = 1이면 신규 삽입) @Query(value = "INSERT INTO likes (user_id, target_type, target_id) " + "VALUES (:userId, :targetType, :targetId) " + "ON DUPLICATE KEY UPDATE id = id", nativeQuery = true) int insertIgnore(Long userId, String targetType, Long targetId); // 실제 insert된 경우에만 count 증가 int affected = likeRepository.insertIgnore(userId, targetType, targetId); if (affected > 0) { likeRepository.incrementCount(targetType, targetId, +1); }
  • INSERT IGNORE 대신 ON DUPLICATE KEY UPDATE를 선택: INSERT IGNORE는 유니크 제약 외에 NOT NULL, FK 위반도 모두 경고로 묻어버리는 문제
  • JDBC URL에 useAffectedRows=true 추가: MySQL Connector/J 기본값은 중복 시에도 found rows=1을 반환해 모든 스레드가 count를 중복 증가시키는 문제 발견 → changed rows 기준으로 전환하여 해결
  • 삭제도 DELETE 영향 row 수가 1인 경우에만 count 감소 — 동일한 멱등 구조 적용
Result
동일 사용자 JMeter 20 VUser 동시 요청 (ramp 1s)유니크 제약 예외 다수→예외 0건, 멱등 처리
  • 중복 요청에도 row 생성과 count 증가가 각각 1회만 반영
  • CountDownLatch 기반 단위 테스트(20 스레드 동시 실행)로 예외 0건, 멱등 처리 검증 통과

1. EC2 서버 장애 감지 자동화 — SSH 메트릭 수집 + 상태 전이 알림

Problem
  • t2.micro에서 Spring Boot + MySQL만으로도 리소스가 빠듯해 Prometheus + Grafana를 EC2에 함께 띄우기 어려워 로컬 PC에서 모니터링

Analyze
  • 단일 시점 gauge 값만으로는 평균 응답시간·에러율 같은 rate 지표를 산출할 수 없음 — 이전 snapshot과의 counter delta 계산이 필요
Action
Railway Container └─ health-check.sh ── SSH ── EC2 actuator/node-exporter/mysqld-exporter ├─ 10분 snapshot 저장 ├─ counter delta 기반 트래픽 지표/응답시간/에러율 계산 ├─ 임계값 기반 상태 전이 감지 (Memory ≥80% WARN, ≥95% CRITICAL 등) └─ Discord Webhook 알림 + runbook 기반 권장 조치

  • 이전 snapshot 대비 counter delta로 트래픽 지표, 평균 응답시간, 에러율을 계산 — 첫 수집처럼 이전 sample이 없는 경우 rate 수치를 계산 대기로 처리해 오탐 방지
  • WARN·CRITICAL·복구 상태 전이 발생 시 Discord Webhook으로 즉시 알림
Result
장애 인지 소요시간수동 확인 (PC 종료 시 불가)→최대 10분 이내 자동 알림

2. 정기 감지·권장 조치·보고 계층 분리로 LLM 비용과 장애 영향 범위 축소

Problem

  • 알림 발생 시 권장 조치도 매번 LLM을 호출하면 비용과 응답 지연이 누적

매번 LLM 호출		 같은 종류·심각도의 알림마다 같은 권장 조치를 반복 생성. 알림 지연 발생 기각
임계값 기반 상태 판단 +
Runbook 우선 +

Action
Detection Layer (Bash, LLM 없음) └─ health-check.sh: 임계값 비교 → 상태 전이 감지 ↓ State Layer (snapshot) └─ /tmp/health-check-alerts/status-current.env ↓ Recommendation Layer (JSON runbook, LLM 없음) └─ JSON runbook: 알림 종류·심각도별 권장 조치 우선 적용 └─ 미매칭 시에만 LLM fallback ↓ Analysis Layer (LLM, 필요 시에만) └─ Daily Summary(9AM) / Full Report(6시간) / 사용자 질문 ↓ Notification Layer └─ Discord Webhook
  • 정기 감지와 기본 장애 알림은 Bash 임계값 기반으로 처리해 정상 반복 체크의 토큰 비용 0원, LLM 장애 시에도 감지와 알림은 계속 동작
  • JSON runbook에 정의한 알림 종류·심각도별 권장 조치를 우선 적용하고, 예외적인 알림만 LLM fallback으로 처리
  • LLM은 사용자 질문, Daily Summary·Full Report 작성, runbook에 직접 절차가 없는 알림의 fallback처럼 해석이 필요한 구간에만 사용해 수집·판단 경로와 분석 경로를 분리
  • Full Report는 helper가 full_report_should_report=true로 판단한 경우에만 LLM 호출, 변화 없으면 생략
Result
정상 상태 토큰 비용매 10분마다 소비→0원 (bash 처리)
권장 조치 경로매번 LLM 호출→runbook 우선, 미매칭만 LLM fallback

3. 모니터링 파이프라인 자기 진단 — “서버 장애”와 “감지 불능”을 구분

Problem
  • 모니터링 시스템 자체에 장애가 발생하면(SSH 끊김, exporter 응답 없음 등) 알림이 오지 않아 “정상”으로 오인할 수 있음
  • 오래된 snapshot을 최신 상태로 보고하거나, exporter 누락으로 불완전한 데이터를 그대로 전달할 위험
Analyze
  • SSH 실패, 메트릭 파싱 실패, 필수 메트릭 누락, 스냅샷 쓰기 실패 등 파이프라인 단계별로 장애 유형이 다르므로 유형별 추적이 필요
  • 일시적 네트워크 지연과 실제 장애를 구분하려면 단일 실패로 즉시 알림하면 안 됨 — 연속 실패 카운트 기반 판단 필요
  • snapshot 측정 시각 기반 freshness 판정이 없으면 stale data를 현재 상태로 오인하게 됨
Action
health-check.sh / heartbeat-context.sh ├─ SSH 연결 확인 → 실패 시 연결 실패 카운트 증가 ├─ 메트릭 수집/검증 → 파싱 실패 또는 필수 값 누락 시 resource_parse 카운트 증가 └─ snapshot 저장 → 저장 실패 시 snapshot_write 카운트 증가 [유형별 2회 연속 실패] → "모니터링 장애 알림" snapshot freshness 판정 ├─ <20분: OK ├─ 20~30분: WARN (오래됨) └─ >30분: CRITICAL (수집 중단 가능성) 필수 메트릭/핵심 값 검증 ├─ app/node/mysql metrics 누락 ├─ 핵심 값 비어 있음/0 이하 └─ 위 경우 모두 resource_parse로 처리하고 snapshot 갱신 중단
  • 파이프라인 각 단계의 실패를 유형별로 추적, 동일 유형 2회 연속 실패 시 “모니터링 장애” 알림 발송 — 일시적 네트워크 지연과 실제 장애 구분
  • snapshot freshness 검증으로 오래된 데이터를 현재 상태로 오인하는 문제 방지
  • 필수 메트릭 누락 검증으로 불완전한 snapshot이 리포트에 포함되지 않도록 차단
Result
  • “서버가 죽은 것”과 “모니터링이 죽은 것”을 구분해 알림 — 감지 불능 상태를 정상으로 오인하는 문제 방지
  • 오래되거나 불완전한 데이터가 리포트에 포함되지 않도록 검증 계층 확보

2. GitHub Actions + Jib 기반 CI/CD 자동화

Problem
  • 로컬 빌드, Docker 이미지 생성, Docker Hub push, EC2 재배포까지 수동으로 처리
Action
main push ↓ GitHub Actions └ REST Docs 빌드 └ Jib 이미지 생성 → Docker Hub push ┌ SSH → EC2 docker-compose up -d
  • Jib를 사용해 Dockerfile 없이 이미지 빌드 — 레이어 캐싱으로 빌드 효율 향상

Result
  • 문서 생성, 이미지 빌드, 배포가 하나의 파이프라인으로 연결
  • 수동 배포 절차를 제거해 팀이 기능 개발과 검증에 집중할 수 있도록 지원
